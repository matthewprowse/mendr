**Date:** 2026-05-22

## Summary

The API routes directory is well-structured with no orphaned files and consistent adherence to Next.js App Router conventions. All non-route.ts files have been correctly purged or delegated to `lib/` (providers/handler.ts pattern). Rate limiting is present on nearly all public routes; cron routes all have bearer-token authentication. The main findings are:

1. **HIGH: Missing rate limit on beta-access route** — password-checking endpoint vulnerable to brute force
2. **HIGH: diagnose route at 1555 lines** — contains extractable orchestration logic (prompt composition, quota handling, streaming setup)
3. **MEDIUM: Three cron routes not in vercel.json** — contractor-onboarding, homeowner-followup, homeowner-reengagement (may be manual-only, worth documenting)
4. **MEDIUM: refine route at 507 lines** — substantial inline image/context handling and Gemini orchestration
5. **LOW: Type definitions inline in diagnose route** — interfaces like Provider, ContentPart, ContentMessage should move to types file

All 17 admin routes correctly use `requireAdmin`. All 9 cron routes correctly use `isAuthorizedCronRequest`. No console.log, @ts-ignore, or catch-any patterns found. No imports from components or app/ directories. The API surface is clean.

## Findings

### F1. [HIGH] Missing rate limit: beta-access endpoint vulnerable to brute force — src/app/api/beta-access/route.ts

**Check:** rate-limiting on public routes  
**Severity:** HIGH (password-guessing endpoint)  
**Route:** POST `/api/beta-access`  
**Evidence:** 
- No `checkRateLimit` call in the route handler
- Route accepts a password in JSON body (line 28: `body.password`)
- No rate limiting defined in `rate-limit-config.ts` for this endpoint
- Compares submitted password to `COMING_SOON_PASSWORD` env var (line 34: `if (!submitted || submitted !== expected)`)

**Analysis:** The beta-access gate is meant to control early access via password. Without rate limiting, an attacker can brute-force the password. This is a low-complexity endpoint (no external APIs, no auth layer to lean on), which makes it an obvious target.

**Recommendation:** Add rate limiting before password check:
```typescript
export async function POST(req: NextRequest): Promise<NextResponse> {
    const limited = await checkRateLimit(req, 'betaAccess');
    if (limited) return limited;
    // ... existing logic
}
```
Then add to `rate-limit-config.ts`:
```typescript
// Beta password-gate — generous because legitimate testers only need one call.
betaAccess: {
    windowMs: 60 * 60 * 1000, // 1 hour
    max: 10,
},
```

**Risk if applied:** Low — standard rate-limit pattern, no side effects

**Owner action:** ☐ Approve  ☐ Defer

```typescript
interface HistoryMessage { ... }
```
These are request/response contracts used only within this route, but repeating the pattern in refine/route.ts (lines 41–44 define ContentPart again).

**Lines 278–302: Image normalization logic (70+ lines, should extract to lib)**
```typescript

    if (Array.isArray(imageUrlsField)) { ... }
    if (Array.isArray(imagesField)) { ... }
```

**Lines 650–900+: Prompt construction and provider hydration**

```typescript
    // Quota check (keep brief or extract to lib/diagnosis/quota.ts)
    const quotaCheck = await checkDiagnosisQuota(req, body);
    if (quotaCheck.denied) return quotaCheck.response;
    
    // Delegate all orchestration
    const result = await runDiagnosisFromRequest(req, body);

    return result.response;

```

**Lines 59–80: `imageStringToInlineData` function (22 lines, recursive image fetching)**
```typescript
async function imageStringToInlineData(img: string): Promise<ContentPart | null> {
    if (img.startsWith('data:')) { ... }
    if (img.startsWith('http') && !isAllowedImageUrl(img)) return null;
    try {
        const res = await fetch(img);
        if (!res.ok) return null;
        const mimeType = res.headers.get('content-type') || 'image/jpeg';

        return { inlineData: { data: base64Data, mimeType } };
    } catch { return null; }
}
```

```typescript
const existingArr = (row as { image_urls?: unknown }).image_urls;

const finalImageUrls = merged.slice(0, MAX_DIAGNOSE_IMAGES);
```

**Recommendation:**
Extract shared functions to `lib/diagnosis/`:
1. `image-preprocessing.ts` — `imageStringToInlineData`, `arrayBufferToBase64`
2. `normalize-images.ts` — image merging + ordering logic

**Check:** type definition locations  
**Severity:** LOW-MEDIUM (code smell, not functional issue)  
**Location:** Lines 217–241  

**Evidence:**  
```typescript
interface Provider { ... }
interface ContentPart { ... }
interface ContentMessage { ... }
interface HistoryMessage { ... }
```

These are also defined in `refine/route.ts` (lines 41–44: `interface ContentPart { ... }`).

**Analysis:** Inline interfaces are fine for request-specific contracts (types that are not imported elsewhere).
```typescript
export interface ContentPart {
    text?: string;
    inlineData?: { data: string; mimeType: string };
}

export interface ContentMessage {
    role: 'user' | 'model';
    parts: ContentPart[];
}
```

Then import in both diagnose and refine routes instead of redefining.

**Risk if applied:** None — purely organizational

**Owner action:** ☐ Approve centralization  ☐ Defer
