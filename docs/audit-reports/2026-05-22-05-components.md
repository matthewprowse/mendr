## Summary

The `src/components/` directory is well-organized and hygienically maintained. The 17 top-level components follow naming conventions (kebab-case filenames), use Next.js `Link` for internal navigation, and have no forbidden patterns (catch-any, @ts-ignore, console.log, relative `<a>` links, imports from `src/app/` routes). All components have clear, documented exports. The `ui/` subdirectory contains 31 shadcn primitives with minimal custom configuration (primarily `cn()` utility wrapping); all files conform to shadcn standards. 

The match subdirectory contains one component (`flow-shell.tsx`) that acts as a flow layout primitive. No dead code found. No file-size violations (all top-level components ≤ 233 lines except `providers-map.tsx` at 709 lines, which is intentional: it manages Google Maps initialization, marker rendering, route calculation, and provider filtering in a single component).

**Key finding:** `providers-map.tsx` (709 lines) is the only component approaching the complexity ceiling. It handles map lifecycle, marker updates, directions rendering, and filter state—which it does well—but the JSX return block spans 137 lines and could benefit from extractive refactoring. The component is referenced and active.

**Framer-motion import:** `hero-scan-mockup.tsx` imports framer-motion. This is **not a hot-path route** (landing page mockup), so the flag is informational only. Current usage is minimal (two animated elements).

No custom (non-shadcn) files found in `ui/`. All naming conventions respected. All components actively imported. No lint violations in components.
